为什么js的for in语句有时在遍历数组时,会出现不存在的数组成员?

发表于 2019-04-23

我们都知道,JavaScript语法中的for…in 语句用于遍历数组或者对象的属性（对数组或者对象的属性进行循环操作）。具有相似功能的语句还有forEach()函数、for..of语句（es6特性）以及经典的的普通for循环：

for…in 语句：
`var arr = [1,2,0,3,9];
for (var index in arr){
console.log(arr[index]);
console.log(index);

}`

forEach()函数：

`arr.forEach(function(value,i){

　　console.log(‘forEach遍历:’+i+’–’+value);
})
`

for…of语句：
`for( let i of arr){
console.log(i);

}`

普通for循环：

for ( var i = 0; i <arr.length; i++){
console.log(arr[i]);
}

潜意识里一直认为这几种遍历方式都差不多，在心里把它们划等号了。
最近做项目遇到了这样的情况，在使用for…in语句遍历一个数组时，不知什么原因，总是会多出一个成因。大概就是如下这样非常简单的的一段代码里：
var tableHead = [
{column: “DEPO_NAME”, name: “企业名称”}，
{column: “EST_DATE”, name: “企业成立日期”}
];
for(var i in tableHead){
var headName = tableHead[i].name;
console.log(headName);
}

按理说，这个循环体最多循环两次就结束了，但是在测试过程中，这个循环体实际上执行了三次。
为了弄清楚是什么原因，在浏览器中下断点来查看什么情况。当第三次进入循环体时，注意到如下图所示的情况：

what？？ i=”remove”？？！！

我寻思这数组里也没有remove这个成员呀！
无奈之下对这个问题进行了查找，在别人博客里找到了答案。

for…in 并不能够保证返回的是按一定顺序的索引，但是它会返回所有可枚举属性，包括继承属性。

给原型添加属性之后，默认情况下枚举，会把原型属性一并输出，如下所示。由于它总是会访问该对象的原型,看下原型上是否有属性,这在无意中就给遍历增加了额外的压力。
function fun4(){
var a=[1,2,3,4,5];
Array.prototype.age=13;
for(var i in a){
console.log(“a[“+i+”]: “+a[i]);
}
}
//outuput:
//a[0]: 1
//a[1]: 2
//a[2]: 3
//a[3]: 4
//a[4]: 5
//a[age]: 13

明白这一点之后，立刻回到项目中使用浏览器再调试一下。果不其然，我们的数组的原型属性中包含了“remove”字段：

到此，问题有了答案，很有可能的是项目采用的某个前端框架对Array对象进行了扩展，使得remove这个扩展属性被for…in迭代了出来。
另外，根据网上其他的资料介绍，for..in得到的index通常是字符串型，而普通for循环的索引是数值型的，看来以后遍历数组还是采用最原始的普通for最保险。

找不到libaio.so.1文件：记fio测试过程中的一次踩坑经验

发表于 2018-11-29

找不到libaio.so.1文件：记fio测试过程中的一次踩坑经验

起因

最近想要做一个关于openstack虚拟机性能折损的测试，就是让openstack创建出来的虚拟机与同等配置的物理机进行性能对比，大致估计一下虚拟化产生的性能损耗。具体的思路就是在两台机器上跑相同的应用程序或跑分软件，然后做结果对比。在测试IO性能的时候，查到fio工具似乎比较好用，于是执行apt install fio命令下载安装试试，ok这里没问题。

接下来，执行fio命令，看下是否安装成功。然后问题就来了，系统弹出一个错误提示：

error while loading shared libraries: libaio.so.1: cannot open shared object file: No such file or directory

第一反应，赶紧谷歌搜一下（毕竟百度不靠谱）libaio.so.1这个文件是个啥，为什么会缺失。但是找了很久，都没有看到比较好的解决办法，无奈只能自己想办法了。

系统环境

虚拟机和物理机都是Ubuntu16.0_X64_Server版本.

解决思路

既然物理机上找不到，那就重新开一台纯净的虚拟机上找找看，说不定是安装过程中除了什么问题导致这个文件被删了呢。

在新创建的虚机上执行 sudo find / -name libaio.so.1

什么也没找到,说明这个文件ubuntu系统本身是没有这个库文件的.
那就安装一下fio试试吧,执行apt install fio

安装结束，再搜索一下看看：

这次有结果了！
在/lib/x86_64-linux-gnu目录下也确实看到了libaio.so.1这样一个文件

到这里基本已经有头绪了，应该是由于物理机上安装fio的过程中出现什么问题了，导致相关依赖的库文件没有安装上。既然如此那就重新安装一遍把！

# 清除fio安装包及其相关配置
apt purge fio

另外，与fio相关的依赖包也要重装（apt install fio时有提示过）：

# 清除依赖包
apt purge libaio1 libboost-iostreams1.58.0 libboost-random1.58.0 libboost-system1.58.0 libboost-thread1.58.0 libibverbs1 libnspr4 libnss3 libnss3-nssdb librados2 librbd1 librdmacm1

清除完毕后，执行apt install fio重新安装，测试一下fio命令，大功告成！

Ubuntu16.0环境下快速部署基于Ocata版本的devstack

发表于 2018-08-26

参照网上教程搭建devstack过程中遇到了不少坑,在这里记录一下,说不定能帮助一下其他人呢哈哈!!

参考资料: https://blog.csdn.net/pfztab/article/details/78632393

1. 环境信息

Ubuntu OS(Ubuntu Server 16.04.4 LTS X64版本)
下载地址: http://ftp.sjtu.edu.cn/ubuntu-cd/16.04.4/ubuntu-16.04.4-server-amd64.iso
DevStack版本：Ocata（安装时指定）
OpenStack版本：Ocata（安装时指定）
内存: 6GB(内存理论上是越大越好,但是实际情况我们可能没有那么多的内存,推荐最好6GB以上)
硬盘: 40GB

2. 准备工作

2.1 设置ubuntu源地址

ubuntu默认源速度慢而且不稳定,我们默认配置Ubuntu的apt的国内源.
首先对 /etc/apt/sources.list 文件进行备份:

cp /etc/apt/sources.list /home/sources.list.bak

使用阿里apt源:

vim /etc/apt/sources.list

将里面的内容全部替换为如下内容:

#注释内容为代码源文件,如果不看源代码的话,可以注释
#deb-src http://mirrors.aliyun.com/ubuntu/ xenial main restricted universe multiverse
#deb-src http://mirrors.aliyun.com/ubuntu/ xenial-updates main restricted universe multiverse
#deb-src http://mirrors.aliyun.com/ubuntu/ xenial-backports main restricted universe multiverse
#deb-src http://mirrors.aliyun.com/ubuntu/ xenial-security main restricted universe multiverse
#deb-src http://mirrors.aliyun.com/ubuntu/ xenial-proposed main restricted universe multiverse

deb http://mirrors.aliyun.com/ubuntu/ xenial main restricted universe multiverse
deb http://mirrors.aliyun.com/ubuntu/ xenial-updates main restricted universe multiverse
deb http://mirrors.aliyun.com/ubuntu/ xenial-backports main restricted universe multiverse
deb http://mirrors.aliyun.com/ubuntu/ xenial-security main restricted universe multiverse
deb http://mirrors.aliyun.com/ubuntu/ xenial-proposed main restricted universe multiverse

2.2 更新 软件包

apt-get update

2.3 安装git

ubuntu已集成,跳过这一步

2.4 安装和配置pip

安装:

apt-get install python-pip

配置:
在/root目录下，查看是否存在“.pip”目录：
若不存在，则手动创建：

mkdir .pip

在.pip目录下创建一个pip.conf的配置文件

touch pip.conf

打开该配置文件,将pip的源信息设置为豆瓣源:

[global]
timeout = 6000
index-url = http://pypi.douban.com/simple/
[install]
trusted-host = pypi.douban.com

3. 开始安装

3.1 下载DevStack代码:

进入/home目录，执行以下命令，下载DevStack(ocata版本)：

git clone http://git.trystack.cn/openstack-dev/devstack.git -b stable/ocata

3.2 配置

创建stack用户,在/home/devstack/tools目录下，执行命令：

./create-stack-user.sh

设置用户密码: 

passwd stack

切换到stack用户,并修改/home/devstack目录下的文件为stack用户权限: 

# 切换用户
    su root
# 执行命令并 修改
    chown -R stack:stack /home/devstack
# 由于后续我们调试需要用到screen命令，因此也同样需要修改对应目录权限：
    chmod 777 /dev/pts/0

配置stack用户的pip源:

# 切换到stack用户,并进入stack的家目录(/opt/stack)
su stack
cd ~
#与在root用户的操作相似,在家目录创建.pip文件夹和pip.conf文件(步骤省略)

注意，如果你是从root用户目录下拷贝过来的，需要查看对应目录及配置文件的权限是否正确，这里都应是stack用户权限的。

设置DevStack配置文件(local.conf):

# 进入/home/devstack目录
cd /home/devstack

# 从./samples子目录中中拷贝一份local.conf文件到当前目录下（即/home/devstack）。
cp /home/devstack/samples/local.conf  /home/devstack/local.conf

打开该文件，进行编辑,替换内容如下:
[[local|localrc]]

# use TryStack git mirror

GIT_BASE=http://git.trystack.cn

NOVNC_REPO=http://git.trystack.cn/kanaka/noVNC.git

SPICE_REPO=http://git.trystack.cn/git/spice/spice-html5.git

#OFFLINE=True

RECLONE=False

# Define images to be automatically downloaded during the DevStack built process.

DOWNLOAD_DEFAULT_IMAGES=False

IMAGE_URLS="http://download.cirros-cloud.net/0.3.4/cirros-0.3.4-x86_64-disk.img"

DATABASE_PASSWORD=admin

ADMIN_PASSWORD=admin

SERVICE_PASSWORD=admin

RABBIT_PASSWORD=admin

# ``HOST_IP`` and ``HOST_IPV6`` should be set manually for best results if
# the NIC configuration of the host is unusual, i.e. ``eth1`` has the default
# route but ``eth0`` is the public interface.  They are auto-detected in
# ``stack.sh`` but often is indeterminate on later runs due to the IP moving
# from an Ethernet interface to a bridge on the host. Setting it here also
# makes it available for ``openrc`` to include when setting ``OS_AUTH_URL``.
# Neither is set by default.
#HOST_IP=w.x.y.z
#HOST_IPV6=2001:db8::7
HOST_IP=192.168.80.130

# Logging
# -------

# By default ``stack.sh`` output only goes to the terminal where it runs.  It can
# be configured to additionally log to a file by setting ``LOGFILE`` to the full
# path of the destination log file.  A timestamp will be appended to the given name.
LOGFILE=/opt/stack/logs/stack.sh.log

# Old log files are automatically removed after 7 days to keep things neat.  Change
# the number of days by setting ``LOGDAYS``.
LOGDAYS=2

# Nova logs will be colorized if ``SYSLOG`` is not set; turn this off by setting
# ``LOG_COLOR`` false.
LOG_COLOR=True

# Uncomment these to grab the milestone-proposed branches from the
# repos:
HORIZON_BRANCH=stable/ocata

KEYSTONE_BRANCH=stable/ocata

NOVA_BRANCH=stable/ocata

NEUTRON_BRANCH=stable/ocata

GLANCE_BRANCH=stable/ocata

CINDER_BRANCH=stable/ocata

#keystone

KEYSTONE_TOKEN_FORMAT=UUID

所有这些工作都做完之后，重启一下虚拟机。

4. 启动安装程序

重启完毕，进入/home/devstack目录，执行命令,进行devstack的安装
./stack.sh
这里可能要等待非常长的时间,因为网络不稳定的原因,经常安装到一半的时候,提示安装失败.
有时候可能要反复安装四五次才能成功(亲身经历!!).
万一安装过程中而失败, 一定要先使用./unstack.sh命令清除安装, 然后./stack.sh重试

如果看到如下显示信息，恭喜你,安装完成!!

根据提示信息我们可以来到登录界面,如下图:

到此devstack的安装就算完成了.

如果是在虚拟上部署的devstack和openstack项目, 启动之后, 如果关闭虚拟机, 下次重新启动虚拟机的时候需要重新启动服务,步骤如下:
使用stack账号登录unbuntu系统, 通过命令 screen -c stack-screenrc 重启所有服务
(必须使用stack账号直接登录,如果通过root账号登录系统,再通过su stack的方式切换到stack账号来执行命令的话,可能会出现screen Cannot open your terminal ‘/dev/pts/1’的问题!!!)
